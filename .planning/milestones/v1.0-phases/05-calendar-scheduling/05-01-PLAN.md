**src/lib/upload-post.ts -- Add publishTextPost function:**

Add a new exported function `publishTextPost` that:
- Accepts `{ uploadPostUsername: string, platforms: string[], content: string }`
- Uses `getBreaker('upload-post').call(...)` for circuit breaker protection
- Sends POST to `${BASE_URL}/upload_text` using FormData (multipart/form-data)
- Sets fields: `user` = uploadPostUsername, repeated `platform[]` for each platform, `title` = content
- Uses `authHeaders()` for auth
- Returns the parsed JSON response
- Does NOT pass `scheduled_date` (project owns scheduling, publishes immediately)
- Store whatever ID field is returned (request_id or job_id) for analytics correlation

**src/lib/publish.ts -- Create publishDuePosts:**

Create the publish module with:
1. `__publishRunning` mutex on globalThis to prevent overlapping cron ticks
2. `publishDuePosts()` async function that:
   a. Checks mutex; if running, log skip and return
   b. Sets mutex = true in try/finally
   c. Queries posts WHERE status='scheduled' AND scheduled_at <= now (ISO string comparison)
   d. For each post, queries its postPlatforms WHERE status='pending' AND (retryAt IS NULL OR retryAt <= now)
   e. For each eligible platform, looks up the socialAccount for that brand+platform to get uploadPostUsername
   f. Calls `publishTextPost()` with the platform's content (or post.content if platform content is null)
   g. On success: update postPlatforms.status = 'published', set postPlatforms.requestId from response
   h. On failure: increment failureCount, set retryAt = now + 5 minutes if failureCount < 3, set status = 'failed' if failureCount >= 3
   i. After processing all platforms for a post: check if ALL postPlatforms are 'published' -> set posts.status = 'published' and posts.publishedAt = now. If ALL are 'failed' -> set posts.status = 'failed'. Otherwise leave as 'scheduled'.
   j. Log each publish attempt to activityLog (type: 'publish', level: 'info' for success, 'error' for failure) with brandId
   k. Catch per-platform errors individually so one failure does not stop others
   l. Also call `incrementFailureCount(accountId)` from accounts actions on platform failure (already exists for Phase 5 use)

**src/lib/cron.ts -- Add publish cron:**

Inside `initCron()`, before the `__cronRegistered = true` line, add:
```
cron.schedule('* * * * *', async () => {
  try {
    const { publishDuePosts } = await import('./publish')
    await publishDuePosts()
  } catch (err) {
    console.error('[cron] auto-publish failed:', err)
  }
})
```
Update the console.log to include 'publish' in the registered jobs list.

**src/app/actions/schedule.ts -- Create server actions:**

1. `schedulePost(postId: number, scheduledAt: string)` -- Updates posts.status to 'scheduled' and posts.scheduledAt. Revalidates /calendar path.

2. `reschedulePost(postId: number, newScheduledAt: string)` -- Updates posts.scheduledAt, updates posts.updatedAt. Used by calendar drag-and-drop. Revalidates /calendar.

3. `getSchedulingSlots(brandId: number)` -- Returns all scheduling_slots rows for the brand, ordered by platform, hour, minute.

4. `saveSchedulingSlots(brandId: number, slots: Array<{ platform: string; hour: number; minute: number }>)` -- Deletes all existing slots for brandId, inserts new ones. Transaction pattern.

5. `scheduleToNextSlot(postId: number, brandId: number, platform: string)` -- Finds the next available slot after now for this brand+platform, applies +/-15 min jitter (`Math.random() * 30 - 15` minutes), sets scheduledAt and status='scheduled'. If no slots configured, returns an error message.

6. `getCalendarEvents(brandId?: number)` -- Queries posts with status in ['scheduled', 'published', 'failed'] joined with postPlatforms to get platform info. Returns array shaped for FullCalendar: `{ id, title, start, color, extendedProps: { status, platforms, brandName, contentType } }`. If brandId provided, filter by brand; otherwise return all brands.

All actions should be marked 'use server' and import from '@/db' and '@/db/schema'.
