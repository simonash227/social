Present the following verification checklist to the user for human testing of all SCHED requirements:

1. **Calendar Views (SCHED-01):**
   - Visit http://localhost:3000/calendar
   - Verify month view renders as default with grid of days
   - Click "week" button in toolbar -- verify week view with time slots appears
   - Click "month" button to switch back

2. **Platform Colors and Status (SCHED-03):**
   - If posts exist: verify events show colored platform dots and status indicators
   - If no posts: create a test post via /brands/[id]/generate, save as draft, then schedule it

3. **Drag-and-Drop (SCHED-02):**
   - Drag a scheduled event to a different day
   - Verify the event stays on the new day after release
   - Check the database or activity log to confirm scheduled_at updated

4. **Slot Configuration (SCHED-04):**
   - Select a brand using the brand filter on the calendar page
   - Find the "Configure Schedule" section
   - Add a time slot for a connected platform (e.g., 09:00 for Twitter)
   - Click Save -- verify it persists on page refresh

5. **Jitter (SCHED-05):**
   - Schedule a post to the next available slot
   - Inspect the scheduled_at value -- it should be offset by up to +/-15 minutes from the exact slot time

6. **Auto-Publish (SCHED-06):**
   - Schedule a post with scheduled_at set to 1 minute from now (or in the past)
   - Wait for the cron to fire (check server console for "[cron]" log)
   - Verify the post status changes to published (or failed if no valid Upload-Post account)

7. **Retry Logic (SCHED-07):**
   - If possible, trigger a publish failure (invalid account or disconnected platform)
   - Verify failureCount increments in the database
   - Verify retryAt is set to 5 minutes later
   - After 3 failures, verify post status becomes 'failed'

8. **Dark Mode:**
   - Verify the entire calendar page uses dark theme (no white/light backgrounds)
