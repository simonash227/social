  <name>Task 1: Auto-generate module with automation level routing</name>
  <files>src/lib/auto-generate.ts, src/app/actions/generate.ts</files>
  <action>
1. Create `src/lib/auto-generate.ts` exporting `autoGenerate(): Promise<void>`. Implementation:

   a. **Mutex guard:** Use `globalThis.__autoGenerateRunning` pattern (same as publish.ts).

   b. **Query eligible entries:** Find feedEntries where:
      - `relevanceScore IS NOT NULL` (has been scored)
      - `relevanceScore >= feedSource.relevanceThreshold` (or >= 6 if threshold is null)
      - `processedAt IS NULL` (not yet processed)
      - Join to feedSources to get brandId, targetPlatforms, relevanceThreshold
      - Join to brands to get automationLevel
      - Filter: `brands.automationLevel != 'manual'` (skip manual brands)
      - Limit to 5 entries per run (avoid context exhaustion)

   c. **Mark entries as processing immediately:** Set `processedAt = new Date().toISOString()` BEFORE starting generation to prevent double-pick on overlapping ticks (research pitfall 4).

   d. **Content mix check (FEED-07):** For each entry, before generating, check if the same sourceUrl was used to create a post for this brand in the last 48 hours. If so, skip (log info to activityLog: 'Skipped -- topic already covered within 48h'). Also use the entry title to do a basic keyword overlap check: extract top 3 keywords from entry title, query recent post titles for same brand. If >50% keyword overlap, skip.

   e. **Extract content:** Call `extractFromUrl(entry.url)` to get source text.

   f. **Determine target platforms:** Use feed's `targetPlatforms`. If null/empty, query socialAccounts for all connected platforms on the brand.

   g. **Run quality pipeline:** Call `generateContent(brandId, platforms, sourceText, entry.url)` then `refineAndGate(brandId, generationResult)`.

   h. **Automation level routing (FEED-08, FEED-09):**
      - `semi`: Always save as draft. Call a new helper `saveAsAutoPost()` which inserts posts with status='draft'.
      - `mostly`: If ALL platform qualityScores >= 7, auto-schedule. If any < 7 but not discarded, save as draft. Discard platforms with discarded=true.
      - `full`: Auto-schedule everything with qualityScore >= 5. Discard platforms scoring < 5.

   i. **Auto-scheduling with spam guard:** For each platform to schedule:
      - Call `checkSpamGuard(brandId, platform, entry.url)`. If not allowed, save as draft instead (log reason).
      - If allowed, call `scheduleToNextSlot(postId, brandId, platform)`.
      - **Cross-platform stagger (SPAM-03):** When scheduling multiple platforms from same entry, add 30-60 min random offset between platforms. After scheduling first platform, for subsequent platforms, find the scheduled time of the first and ensure the next is at least 30 min later.

   j. **Hashtag enforcement (SPAM-07):** After quality pipeline produces content, validate hashtag counts per platform:
      - Count hashtags (words starting with #) in the content.
      - Platform limits: X/twitter: 0-3, instagram: 5-15, linkedin: 3-5, tiktok: 3-5.
      - If over limit: trim hashtags from end until within range.
      - If under minimum (instagram): do NOT add -- the content was generated without them, adding random ones would be spam.
      - Apply this check inside `enforceHashtags(platform: string, content: string): string` helper.

   k. **saveAsAutoPost helper:** Similar to saveGeneratedPosts but:
      - Does NOT call redirect() (not a server action context).
      - Accepts status parameter ('draft' | 'scheduled').
      - Sets feedEntryId on the post for traceability.
      - Returns the post ID for scheduling.

   l. **Log all activity:** Each auto-generated post logs to activityLog with type='auto_generate', including source URL, platforms, quality scores, and final status (draft/scheduled/discarded).

2. In `src/app/actions/generate.ts`: The existing generateContent() and refineAndGate() are marked 'use server' but can be called from server-side code directly. However, saveGeneratedPosts calls redirect() which will throw in non-request context. The auto-generate module should use its own `saveAsAutoPost` helper (defined in auto-generate.ts) instead of calling saveGeneratedPosts. No changes needed to generate.ts.

If they DON'T use server-action-only APIs (beyond the 'use server' directive), they can be imported directly since auto-generate.ts runs server-side. Check the code and handle accordingly -- if revalidatePath is called inside these functions, that call is safe in server-side context (it's a no-op outside request context in Next.js).
