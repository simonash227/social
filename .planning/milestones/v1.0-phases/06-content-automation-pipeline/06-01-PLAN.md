4. Create `src/lib/spam-guard.ts` exporting `checkSpamGuard(brandId: number, platform: string, sourceUrl?: string): Promise<{ allowed: boolean; reason?: string }>`. Implementation:

   a. **Warmup cap (SPAM-04):** If brand.warmupDate exists, compute weeks since warmupDate. Week 0 (0-6 days) = cap 1/day, week 1 (7-13 days) = cap 2/day, week 2+ = null (no warmup cap). Count today's posts (status IN 'published','scheduled') for this brand+platform. If count >= warmupCap, return `{ allowed: false, reason: 'warmup cap N/day' }`.

   b. **Daily platform cap (SPAM-01):** Use constants: `{ twitter: 5, x: 5, instagram: 3, linkedin: 2, tiktok: 3 }`. Count today's posts (status IN 'published','scheduled') for brand+platform. IMPORTANT: count both scheduled AND published posts to avoid over-scheduling (see research pitfall 5).

   c. **Minimum 1-hour gap (SPAM-02):** Query the most recent post (by scheduledAt or publishedAt, whichever is later) for brand+platform. If less than 60 minutes ago, return not allowed.

   d. **Cross-platform stagger (SPAM-03):** Accept optional `sourceUrl` param. If sourceUrl provided, query posts with same sourceUrl where scheduledAt or publishedAt is within last 60 minutes across ANY platform for this brand. Return not allowed if found. (The auto-generate module will add 30-60 min offsets when scheduling multiple platforms.)

   e. **Topic dedup (SPAM-05):** If sourceUrl provided, check if any post with this sourceUrl was created in the last 48 hours for this brand. Return not allowed if duplicate found.

   f. **Link ratio (SPAM-06):** Count posts with non-null sourceUrl in the last 7 days for this brand. Count total posts in same window. If link ratio > 0.35 (35%), return not allowed with reason 'link ratio exceeded'.

   Helper functions (private):
   - `countTodayPosts(db, brandId, platform, todayStr)` -- counts posts with status IN ('published', 'scheduled') AND date portion of (scheduledAt OR publishedAt) = todayStr, joined to postPlatforms for platform filter.
   - `getLastPostTime(db, brandId, platform)` -- returns the most recent scheduledAt or publishedAt for brand+platform.

   All queries use drizzle-orm with the existing schema. Import getDb from '@/db'.

<task type="auto">
  <name>Task 2: Feed polling module with Haiku relevance scoring</name>
  <files>src/lib/feed-poll.ts, next.config.ts</files>
  <action>
1. Install rss-parser: `npm install rss-parser`

2. Add `'rss-parser'` to `serverExternalPackages` array in `next.config.ts` (alongside existing better-sqlite3, node-cron, pdf-parse, openai).

3. Create `src/lib/feed-poll.ts` exporting `pollFeeds(): Promise<void>`. Implementation:

   a. **Mutex guard:** Use `globalThis.__feedPollRunning` pattern (same as publish.ts).

   b. **Query enabled feeds:** `db.select().from(feedSources).where(eq(feedSources.enabled, 1)).all()`

   c. **Iterate feeds sequentially** (not Promise.all -- to avoid overwhelming external servers). For each feed, call `pollSingleFeed(feed)` wrapped in try/catch.

   d. **pollSingleFeed(feed):**
      - Create rss-parser instance: `new Parser({ timeout: 10_000, headers: { 'User-Agent': 'SocialContentEngine/1.0' } })`  (can be module-level singleton).
      - Call `parser.parseURL(feed.url)` to fetch the feed.
      - Collect new entries: for each `item` in `parsed.items`, extract `url = item.link ?? item.guid`. Skip if no url. Try inserting into feedEntries with `feedSourceId`, `url`, `title: sanitizeText(item.title ?? '')`. Catch UNIQUE constraint violation = already seen, skip. Collect successfully inserted entries (with their assigned IDs) into a `newEntries` array.
      - **Batch Haiku scoring (max 20 per poll):** If newEntries.length > 0, take first 20, call `scoreBatch(brand, newEntries)`. This makes ONE Haiku API call scoring all entries at once (not N+1 -- see research pitfall 1).
      - On success: reset `consecutiveFailures` to 0 for this feed.
      - On failure (catch block): increment `consecutiveFailures`. If count >= 10, set `enabled = 0` (FEED-10). Log to activityLog with type='feed_poll', level='error'.

   e. **scoreBatch(brand, entries):**
      - Check `checkAiSpend()` first. If over limit, skip scoring (entries remain with null relevanceScore for next poll).
      - Build prompt listing all entry titles+URLs, asking for JSON array `[{ "id": N, "score": 7 }]`.
      - Call Anthropic via circuit breaker using `getModelConfig().filter` (Haiku).
      - Parse response, update each feedEntry's `relevanceScore` in DB.
      - Log AI spend via `logAiSpend()`.
      - If parse fails, log warning but don't crash -- entries will be re-scored on next poll if relevanceScore is still null.

   f. **Mark relevant entries for processing (FEED-05):** After scoring, entries with `relevanceScore >= feed.relevanceThreshold` (default 6 if null) are candidates for auto-generate. The auto-generate module (Plan 02) will query these. No action needed here beyond storing the score.

   All four feed types (RSS, YouTube channel RSS, Reddit subreddit RSS, Google News RSS) work through `parser.parseURL()` with no special handling needed.
  </action>
