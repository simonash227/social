# Phase 8: Schema Foundation - Research

**Researched:** 2026-03-19
**Domain:** Drizzle ORM additive SQLite migrations — adding tables and columns without breaking existing rows
**Confidence:** HIGH

## Summary

Phase 8 is entirely a database migration task. The goal is to add three new tables (`brandLearnings`, `promptTemplates`, `commentSuggestions`) and new columns on three existing tables (`posts`, `brands`, `postAnalytics`) so all v2.0 features have the schema they need. No application logic changes. No UI changes. The only risk is a migration that breaks existing data or causes the v1.0 automation pipeline to crash.

The existing project uses Drizzle ORM (v0.45.1) with drizzle-kit (v0.31.9) on SQLite via better-sqlite3. Migrations are applied automatically at startup in `src/db/index.ts` via `migrate()`. The pattern for this project is: update `src/db/schema.ts` first, then run `npx drizzle-kit generate` to produce the SQL migration file, then commit both together. The migration runs on next server start — no manual `migrate` command needed for production.

The critical constraint is safety of existing rows. All new columns on existing tables must either be nullable or carry a non-null default. The architecture research document specifies the exact values: `enableVariants` (default 0), `learningInjection` (default 1), `lastLearningRunAt` (nullable), `recycledFromPostId` (nullable), `variantGroup` (nullable), `variantOf` (nullable), `repurposeChainId` (nullable), `promptTemplateId` (nullable), `activeLearningIds` (nullable). No existing rows will be touched; the migration is purely additive.

**Primary recommendation:** Update `schema.ts` with all new tables and columns, run `npx drizzle-kit generate` to produce a single migration file `src/db/migrations/0008_v2_schema.sql`, verify the generated SQL matches the spec, then confirm the v1.0 automation pipeline completes a full cycle without errors.

## Architecture Patterns

### How Migrations Work in This Project

Migrations are applied automatically. The db singleton in `src/db/index.ts` calls `migrate(_db, { migrationsFolder: ... })` on first connection. Any `.sql` file in `src/db/migrations/` not yet recorded in `_journal.json` is applied in order. On Railway, this means the migration runs on the next deploy's cold start.

```
src/db/schema.ts  ─── npx drizzle-kit generate ──▶  src/db/migrations/0008_v2_schema.sql
                                                              │
                                                    (auto-applied at startup via migrate())
```

### Pattern 1: Additive Column Addition — Nullable or Default

All new columns on existing tables must be safe for existing rows. In SQLite, `ALTER TABLE ADD COLUMN` is safe when the column is nullable or has a `DEFAULT`. Drizzle-kit generates the correct SQL automatically when the schema is written correctly.

**Safe pattern for existing tables:**
```typescript
// Source: direct codebase inspection — see 0006_feed_automation.sql
// This is exactly how v1.0 added columns to brands and posts mid-project

// Nullable FK (set by future code, not at migration time):
recycledFromPostId: integer('recycled_from_post_id').references(() => posts.id),

// Integer column with default (existing rows get the default automatically):
enableVariants: integer('enable_variants').notNull().default(0),

// Nullable text (existing rows get NULL automatically):
lastLearningRunAt: text('last_learning_run_at'),
```

The generated SQL will be `ALTER TABLE brands ADD COLUMN enable_variants integer DEFAULT 0 NOT NULL;` — SQLite backfills the default for all existing rows.

### Pattern 2: JSON Array/Object Columns in Drizzle + SQLite

The project stores JSON data as `text` with `{ mode: 'json' }` and a TypeScript generic. This is the existing pattern used for `topics`, `dosList`, `platformNotes` on brands. New columns that store JSON arrays follow the same approach.

```typescript
// Source: src/db/schema.ts (existing pattern)
// supportingPostIds: array of post IDs that validate this learning
supportingPostIds: text('supporting_post_ids', { mode: 'json' }).$type<number[] | null>(),

// activeLearningIds: IDs of learnings active during this post's generation
activeLearningIds: text('active_learning_ids', { mode: 'json' }).$type<number[] | null>(),
```

### Pattern 3: Self-Referential FK in Drizzle

The `posts` table gains two columns that reference `posts.id` itself (`recycledFromPostId`, `variantOf`). Drizzle handles this with a lambda reference to avoid circular dependency at module load time.

```typescript
// Source: ARCHITECTURE.md — Pattern 4: Variant Winner/Loser Tracking
// Self-referential FK pattern — use lambda to avoid circular ref
recycledFromPostId: integer('recycled_from_post_id').references(() => posts.id),
variantOf:          integer('variant_of').references(() => posts.id),
```

### Anti-Patterns to Avoid

- **NOT NULL without default on existing table:** `ALTER TABLE posts ADD COLUMN some_col text NOT NULL` fails in SQLite if there are existing rows and no DEFAULT is specified. Always add `.default(value)` or make nullable.
- **Referencing new table in migration before it's created:** drizzle-kit handles ordering automatically; do not manually write SQL for new tables — let the generator handle order.
- **Multiple migration files for one phase:** All Phase 8 schema changes should land in a single migration file (`0008_v2_schema.sql`). One migration, one commit, one deploy.
- **Editing generated SQL files after generation:** Never modify generated `.sql` files manually. If the spec changes, update `schema.ts` and regenerate. The generator diffs against the snapshot, not the SQL.

## Common Pitfalls

### Pitfall 1: Self-Referential FK Requires Lambda

**What goes wrong:** Writing `references(() => posts.id)` before the `posts` table constant is fully defined causes a TypeScript/runtime error because the variable isn't initialized yet at module evaluation time in non-circular module order.

**Why it happens:** The `posts` table definition itself references `posts` (for `recycledFromPostId` and `variantOf`). JavaScript evaluates module-level statements top-to-bottom; the variable isn't bound yet when the initializer runs.

**How to avoid:** Drizzle's `references()` accepts a callback (thunk) — `() => posts.id` — which is called lazily at query time, not at definition time. This is the correct pattern and already used in the codebase for non-self-referential FKs. The self-referential case is identical in syntax.

**Warning signs:** TypeScript error "Block-scoped variable used before its declaration" or runtime "Cannot access X before initialization".

### Pitfall 2: Drizzle-Kit Generate Without Meta Sync

**What goes wrong:** Running `npx drizzle-kit generate` produces a new SQL file but the `meta/_journal.json` and `meta/0008_snapshot.json` files are not committed alongside `schema.ts`. The next `drizzle-kit generate` run tries to re-diff from the wrong baseline and generates duplicate or conflicting SQL.

**Why it happens:** Developers sometimes `.gitignore` the meta folder or forget to `git add` it.

**How to avoid:** Always commit `schema.ts`, the generated `.sql` file, and the updated `meta/` files together. The `.planning/phases/08-schema-foundation/` git commit should include all four: `schema.ts`, `0008_v2_schema.sql`, `meta/_journal.json`, `meta/0008_snapshot.json`.

**Warning signs:** Running `npx drizzle-kit generate` a second time immediately after the first produces a non-empty diff (it should produce "No changes detected" if schema.ts and meta are in sync).

### Pitfall 3: Migration File Numbering Conflict

**What goes wrong:** drizzle-kit's auto-generated file is named based on the next available index in `_journal.json`. If another branch added a migration file between 0007 and the Phase 8 migration, the numbering collides.

**Why it happens:** Only relevant if multiple branches are active simultaneously. With a single-developer project this is low risk.

**How to avoid:** Always run `git pull` before running `npx drizzle-kit generate` to ensure the local meta folder reflects the latest committed state. Check `_journal.json` to confirm the last index is 7 before generating.

**Warning signs:** `npx drizzle-kit generate` produces a file that starts at index other than 8.

### Pitfall 4: Forgetting to Update `updatedAt` Trigger Pattern on brands

**What goes wrong:** The `brands` table has an `updatedAt` column that the application manually sets to `new Date().toISOString()` on every update. If new columns are added to `brands` and existing update calls don't include them, the `updatedAt` doesn't refresh — minor but worth noting.

**Why it happens:** Not a migration concern — the migration simply adds columns. The update-call behavior is handled in server actions, not in schema.ts.

**How to avoid:** No action needed at migration time. Note for Phase 11 when brand settings update calls are modified.

### Pitfall 5: Railway WAL Mode + Migration at Cold Start

**What goes wrong:** On Railway cold starts (after deploy), two simultaneous incoming requests might both attempt `migrate()` before the singleton is established. The second migrate call may find the migration table locked by the first.

**Why it happens:** The module-level `_db` singleton is initialized on first `getDb()` call. Node.js is single-threaded, but Railway may have multiple instances or health-check requests racing on cold start.

**How to avoid:** This is already mitigated by `PRAGMA busy_timeout = 5000` configured in `openDb()` in `src/db/index.ts`. The second caller will wait up to 5 seconds rather than throwing immediately. Confirmed in codebase — no action needed.

## Open Questions

1. **`status` field on `brandLearnings`**
   - What we know: ARCHITECTURE.md mentions `isActive` (int 0/1) for injection gating and `abTestGroup`/`validatedAt` for A/B lifecycle. REQUIREMENTS.md LEARN-07 says "new learnings start as pending until approved on dashboard."
   - What's unclear: The spec uses `isActive` as the injection gate, but LEARN-07 implies a separate approval state. A learning can be `isActive = 1` (actively injected) but still needs an "approved by human" flag separate from "is it currently on or off."
   - Recommendation: Add a `status` text column (`'pending' | 'approved' | 'rejected'`) alongside `isActive`. New learnings insert with `status = 'pending'` and `isActive = 0`. Human approval sets `status = 'approved'` and `isActive = 1`. Human rejection sets `status = 'rejected'` and `isActive = 0`. This makes the approval gate explicit rather than overloading `isActive`.

2. **`postAnalytics.promptTemplateId` FK — null when no template exists**
   - What we know: `promptTemplates` starts empty. All existing posts will have `promptTemplateId = NULL`. The FK is nullable.
   - What's unclear: Should the default system prompt be inserted as a `promptTemplates` row at migration time (version 1, `isActive = 1`), or is a null `promptTemplateId` acceptable until Phase 9 prompt evolution work?
   - Recommendation: Leave nullable with NULL as valid "no template tracked yet." Phase 9 will insert the first template row when prompt evolution starts. Inserting seed data in a schema migration is an anti-pattern (data migrations belong in application code).
